Registry file:
  ~/.config/llama-server/models.tsv

Format:
  alias<TAB>/full/path/to/model.gguf<TAB>extra_args<TAB>context<TAB>ngl<TAB>batch<TAB>threads<TAB>parallel<TAB>device<TAB>notes

Example:
  gemma4-e4b-q8<TAB>/absolute/path/to/Gemma-4-E4B-Q8_K_P.gguf<TAB>--mmproj /absolute/path/to/mmproj-Gemma-4-E4B-f16.gguf
  qwen35-9b-q8<TAB>/absolute/path/to/Qwen3.5-9B-Q8_0.gguf<TAB><TAB>65536<TAB>999<TAB>128<TAB>16<TAB>1<TAB><TAB>single-client coding preset

Runtime defaults live in:
  ~/.config/llama-server/defaults.env

Useful defaults:
  LLAMA_SERVER_CONTEXT=128000
  LLAMA_SERVER_NGL=999
  LLAMA_SERVER_DEVICE=
  LLAMA_SERVER_PARALLEL=
  GGML_CUDA_ENABLE_UNIFIED_MEMORY=
  LLAMA_SERVER_EXTRA_ARGS=
  LLAMA_MODEL_AUTO_FIT=1

For oversized CUDA contexts that only fail after KV/compute reservation grows,
GGML_CUDA_ENABLE_UNIFIED_MEMORY=1 is an experimental escape hatch. It can let
llama.cpp spill into system RAM, but it may be much slower on discrete GPUs and
usually pairs best with LLAMA_SERVER_PARALLEL=1.

Binary portability:
  llama.cpp source is portable. Built binaries are not universal.
  CUDA, Vulkan, Metal, and CPU builds are backend-specific and may also be
  platform- or architecture-specific.
  llama-model only auto-accepts bundled binaries that pass an explicit host
  compatibility check.
  If no safe runtime is available, run:
    llama-model build-runtime --backend auto
  or from a repo checkout:
    ./scripts/build-llama-server.sh --backend auto

Context MCP repair:
  If doctor reports context_mode_mcp_dist: no or the dashboard shows Context
  missing_dist, run:
    llama-model context-mcp build
  Then restart the LMM gateway and dashboard so routed-full mode can invoke the
  rebuilt Context MCP bundle.

For single-client coding harnesses, a good optional tweak is:
  LLAMA_SERVER_PARALLEL=1

That does not increase context. It simply reduces slot count so the server is more deterministic for one active client.

Startup mitigation:
  switch/restart default to guarded GPU/RAM-aware auto-fit. When llama.cpp hits known CUDA/KV-cache/projector startup failures, llama-model prints the category and may retry with safer runtime-only settings. High system RAM can enable slower hybrid CPU/GPU fallback when VRAM is tight, but it is not treated as extra VRAM. Use --no-auto-fit to disable this for a launch, --require-mmproj to prevent text-only projector fallback, or --force-mmproj only when you intentionally accept a filename-family mismatch.

Dashboard:
  Run llama-model-web or launch "LLM Model Manager" from the desktop or app menu.
  The browser UI can switch models, scan/import local GGUFs, search cached Hugging Face GGUF artifacts, manage serialized downloads, edit presets, save defaults, and inspect runtime/binary diagnostics.
  It also tells you when the registry or defaults files do not exist yet, instead of showing an empty editor with no context.

Remote downloads:
  Search remote GGUF artifacts from the dashboard, then queue downloads through the managed lifecycle. Jobs are persisted under ~/.local/state/llama-server/download-jobs.json and can be inspected or recovered with:
    llama-model download-jobs
    llama-model download-cancel <job_id>
    llama-model download-retry <job_id>

Optional background service:
  The dashboard stays on-demand by default.
  Advanced users can install an opt-in systemd --user service with:
    llama-model dashboard-service install
    llama-model dashboard-service enable
    llama-model dashboard-service restart
    llama-model dashboard-service status
    llama-model dashboard-service logs
  This service is never enabled automatically during install.


Client integrations:
  opencode keeps its own config under ~/.config/opencode/opencode.json and state under ~/.local/state/opencode/model.json.
  Successful model switches sync OpenCode by default via LLAMA_MODEL_SYNC_OPENCODE=1. Set it to 0 for manual-only sync.
  Run `llama-model sync-opencode --preset balanced|long-run` to align it with the active LMM gateway and model.
  Routed mode is the default: harness -> LMM gateway -> GlyphOS AI Compute -> local llama.cpp backend. Use `--mode direct` to bypass the gateway and write the backend endpoint directly.
  Manage the gateway with `llama-model gateway start|stop|restart|status|logs`.
  Upgrades migrate legacy LLAMA_MODEL_OPENCODE_GATEWAY_BASE_URL defaults into LLAMA_MODEL_HARNESS_MODE plus LLAMA_MODEL_GATEWAY_HOST/PORT/LOG, and doctor reports route_state when mixed legacy/direct/routed config is detected.
  The long-run preset also writes top-level compaction headroom (`compaction.reserved`) so large tool-heavy sessions compact before write/edit calls run out of context. Override with OPENCODE_COMPACTION_RESERVED=N if needed.

  OpenClaw keeps its provider config under ~/.openclaw/openclaw.json or ~/.openclaw-<profile>/openclaw.json.
  Run `llama-model sync-openclaw` to point it at the routed LMM gateway. Use `--mode direct` for the backend llama.cpp endpoint. Set LLAMA_MODEL_SYNC_OPENCLAW=1 to also update it after successful switches.

  Claude Code keeps settings in ~/.claude/settings.json and expects an Anthropic-style API.
  Run `llama-model sync-claude` and, when needed, start the local bridge with `llama-model claude-gateway start`. Set LLAMA_MODEL_SYNC_CLAUDE=1 to update settings after successful switches.

  GlyphOS AI Compute can target the same local llama.cpp endpoint.
  A bundled public copy lives under integrations/public-glyphos-ai-compute/ in the repo and under ~/.local/share/llama-model-manager/integrations/ after install.
  Run `llama-model sync-glyphos` to align ~/.glyphos/config.yaml with the active local runtime. Set LLAMA_MODEL_SYNC_GLYPHOS=1 to update it after successful switches.
  Context MCP availability is diagnostic today; gateway requests do not yet invoke a stable Context retrieval API for every routed inference.

Legacy UI:
  Set LLAMA_MODEL_UI=zenity if you want to force the old Zenity launcher.
